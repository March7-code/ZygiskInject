### Connect 模式更隐蔽



现在我有了两个项目的完整分析，可以给出详细的对比和改进方案。

------

## 对比分析

### 当前 ZygiskFrida 的隐藏方案

只做了一件事：**maps 隐藏（remapper.cpp）**



```
inject_lib() → xdl_open/dlopen → remap_lib()
```

`remap_lib` 的原理是把每个 segment 用 `mmap` 分配匿名内存，`memmove` 复制数据，再用 `mremap(MREMAP_FIXED)` 原地替换，使 `/proc/self/maps` 中的路径消失。

**缺失的部分：没有 solist 隐藏。**

这意味着：

- `dl_iterate_phdr()` 仍然能枚举到 frida-gadget
- `dladdr()` 仍然能通过地址反查到库名
- 任何遍历 linker solist 的检测工具都能发现它

------

### AndKittyInjector 的隐藏方案

两层隐藏：

| 层              | 方法                           | 效果                             |
| --------------- | ------------------------------ | -------------------------------- |
| maps 隐藏       | `mremap` 匿名替换              | 同 ZygiskFrida                   |
| **solist 隐藏** | 直接 patch linker 内存中的链表 | `dl_iterate_phdr`、`dladdr` 失效 |

solist 隐藏的核心步骤：

1. 用 `xdl_dsym` 从 linker 中找 `__dl__ZL6solist` / `__dl__ZL6sonext` 符号地址
2. 读取 solist 链表头指针
3. 动态扫描 soinfo 结构体，找到 `base` 和 `next` 字段的偏移（不硬编码，兼容多版本）
4. 遍历链表找到目标 soinfo 节点
5. `prev->next = target->next`，把节点从链表摘除
6. 如果摘除的是尾节点，同步更新 `sonext`

------

## 改进方案

ZygiskFrida 已经有 xdl 库（两个项目用的是同一套 xdl），所以基础设施完全具备，只需要在 [inject.cpp](vscode-webview://0mho8s0j5kbqcg8chlkvjpioc9bc0di6e9aned38d7h6ao21lleh/module/src/jni/inject.cpp) 的 `inject_lib` 函数里，在 `remap_lib` 之后增加一个 `solist_remove` 步骤。

### 方案一：直接移植 SoInfoPatch 逻辑（推荐）

在 ZygiskFrida 的 `inject.cpp` 中，`remap_lib` 之后增加 solist 摘除逻辑：

**关键点：**

- 用 `xdl_dsym` 查 `__dl__ZL6solist` 和 `__dl__ZL6sonext`（xdl 已有，直接用）
- 动态发现 soinfo 的 `base_offset` 和 `next_offset`，不硬编码（兼容 Android 5~15）
- 摘链表节点后更新 `sonext`

**与 AndKittyInjector 的差异点：**

| 差异              | AndKittyInjector              | ZygiskFrida 改进后           |
| ----------------- | ----------------------------- | ---------------------------- |
| 运行上下文        | ptrace 远程进程               | 进程内（in-process），更简单 |
| 内存读写          | `_kMgr->readMem` / `memPatch` | 直接指针解引用即可           |
| NativeBridge 支持 | 有（x86 模拟 ARM）            | 可选，frida 场景通常不需要   |

因为 ZygiskFrida 是 in-process 注入（Zygisk 模块在目标进程内运行），不需要 ptrace 远程读写，代码会比 AndKittyInjector 简洁很多。

### 方案二：同时加固 maps 隐藏

当前 `remap_lib` 有一个小问题：它从 `/proc/self/maps` 读取路径来匹配，但如果库路径包含特殊字符或路径解析有偏差，可能漏掉某些 segment。可以改为按 `dlpi_addr`（load bias）来精确匹配，而不是按文件名字符串匹配。

### 方案三：隐藏时机优化

当前流程：`dlopen` → `remap_lib`（有窗口期）

改进后：`dlopen` → `solist_remove` → `remap_lib`

先摘 solist 再做 maps 隐藏，缩短被检测到的时间窗口。

------

## 总结

ZygiskFrida 目前只防了 `/proc/maps` 扫描，对 `dl_iterate_phdr` / `dladdr` 类检测完全没有防护。AndKittyInjector 的 solist patch 方案可以直接移植，由于运行上下文是 in-process，实现比原版更简单。两个项目都已经包含 xdl 库，所有依赖都已就位，改动范围主要集中在 [inject.cpp](vscode-webview://0mho8s0j5kbqcg8chlkvjpioc9bc0di6e9aned38d7h6ao21lleh/module/src/jni/inject.cpp) 和新增一个 `solist_patch.cpp/h`。

## 结论：不需要第二个 so

`parse_socket_address` 在 `socket.vala:21-28` 里已经支持 `unix:` 前缀，会解析成 **abstract UNIX domain socket**（如果内核支持）或普通路径 socket。

这意味着你可以让 gadget 用 Connect 模式连到一个 **abstract UNIX socket**，而这个 socket 的另一端不需要是独立进程或独立 so——**可以是注入器本身通过 `socketpair` 或 abstract socket 直接桥接**。

但更重要的发现是：**根本不需要代理 so**，因为有一个更干净的架构：

------

### 最终方案：注入器持有代理，进程内零额外 so



```
┌─────────────────────────────────────────────────────┐
│  目标进程                                            │
│                                                     │
│  gadget.so（你已有隐藏手段）                         │
│    ↓ connect("unix:frida-XXXXXX")                   │
│    ↓ abstract UNIX socket（内核命名空间内，不在文件系统）│
└─────────────────────────────────────────────────────┘
           ↕ 内核 socket buffer，跨进程
┌─────────────────────────────────────────────────────┐
│  注入器进程（你自己的进程，在设备上常驻或按需启动）    │
│                                                     │
│  监听同名 abstract UNIX socket                       │
│    ↓ 接收 gadget 的原始 WebSocket+DBus 流量          │
│    ↓ 加密封装                                        │
│    ↓ 通过 TCP 发给 PC 端控制端                       │
└─────────────────────────────────────────────────────┘
```

**进程内只有一个 so（gadget），代理逻辑在注入器进程里**。

Abstract UNIX socket 的特点：

- 不在文件系统，`ls /proc/<pid>/fd` 看到的是 `socket:[inode]`，看不出是什么
- 名字在内核抽象命名空间里，`ss -xp` 可以看到但名字你可以随机生成
- 跨进程通信，不需要任何共享内存或额外 so

------

### 配置文件由注入器动态生成

注入器每次注入时：

1. 生成一个随机 abstract socket 名，比如 `\0frida-a3f9c2`（`\0` 开头表示 abstract）
2. 在注入器进程里监听这个 socket
3. 生成 gadget 配置文件，写入目标 app 的私有目录：



```json
{
  "interaction": {
    "type": "connect",
    "address": "unix:a3f9c2",
    "port": 0
  }
}
```

1. 注入 gadget，gadget 读配置后主动连过来
2. 注入器拿到连接，做加密后转发给 PC 端

**每次注入 socket 名都不同，无法被静态特征匹配**。

### 注入器进程本身的隐藏

注入器进程是你自己的代码，完全可控：

- 改进程名（`prctl(PR_SET_NAME, "kworker/u:3")`）
- 用 `argv[0]` 伪装（`/proc/<pid>/cmdline` 显示的名字）
- 或者直接让注入器在完成注入后 fork 出一个子进程持有 socket，父进程退出

这比隐藏一个 so 容易得多，而且和 frida 版本完全无关。
